## 1. 배경과 목표

### 1.1 현재 제품 구조

D'Flow에는 이미 다음 실행 정보가 존재한다.

- WBS: 계획된 업무, 일정, 담당팀, 진척률, 선후행 관계, 산출물
- 칸반: WBS 말단 작업을 단계·담당·계산 상태별로 재구성한 화면
- 회의·회의록: 회의 일정, 원문, 결정·액션·기한·리스크 AI 인사이트
- 주간업무: 금주 실적, 차주 계획, 이슈·이벤트 자유 텍스트
- 대시보드: 지연, 기한 임박, 진척 정체, 팀 과부하, 오래된 회의 액션 등의 위험 신호

그러나 현재는 발생한 이슈에 대해 다음 내용을 구조적으로 추적할 공간이 없다.

- 누가 책임지고 있는가
- 언제까지 해결해야 하는가
- 현재 어떤 조치를 하고 있는가
- 해결 여부를 누가 검증했는가
- 어떤 WBS·회의록·주간업무에서 발생했는가
- 같은 이슈가 반복되거나 중복 등록됐는가

### 1.2 제품 방향

이슈관리는 계획 업무를 다시 만드는 두 번째 WBS가 아니다. 계획과 실제 실행 사이에서 발생한 **예외, 장애, 위험, 후속 조치**를 책임자와 기한 중심으로 닫는 공간이다.

```text
계획된 업무            실행 중 발생한 예외             보고·학습
   WBS        →        이슈관리 허브        →      대시보드·주간보고
                        ↑        ↑
                    회의록     위험 신호
```

### 1.3 목표

1. 흩어진 현안·리스크·액션을 프로젝트 단위 실행 큐로 모은다.
2. 모든 이슈에 담당팀, 담당자, 우선순위, 기한, 상태를 부여한다.
3. WBS와 회의록 원문을 근거로 연결해 추적 가능성을 확보한다.
4. 담당자의 해결 선언과 PMO의 종료 확인을 분리한다.
5. 변경·조치·첨부 이력을 보존해 감사 가능한 기록을 만든다.
6. 향후 대시보드, 주간보고, 알림, DK Bot과 연결할 수 있는 도메인을 마련한다.

### 1.4 성공 기준

- 사용자가 수동 또는 원문에서 이슈를 등록하고 담당자·기한을 지정할 수 있다.
- 미해결, 긴급, 기한 초과, 해결 요청 이슈를 즉시 구분할 수 있다.
- 이슈에서 연결된 WBS 행과 회의록 원문 위치로 이동할 수 있다.
- 상태 변경과 주요 필드 변경이 모두 감사 이력에 남는다.
- 같은 회의록 블록이나 위험 신호에서 중복 이슈가 생성되지 않는다.
- AI가 사용자 확인 없이 운영 데이터를 변경하지 않는다.

## 2. 기존 기능과의 역할 경계

| 기능 | 관리 대상 | 상태의 의미 | 이슈관리와의 관계 |
|---|---|---|---|
| WBS | 계획된 작업 | 계획 대비 진척 계산 상태 | 관련 작업을 링크하되 이슈 상태와 동기화하지 않는다. |
| 칸반 | WBS 작업의 재표현 | WBS 실적률에서 계산된 상태 | UI 패턴만 참고하고 도메인·상태 로직은 재사용하지 않는다. |
| 이슈관리 | 계획 밖 현안·리스크·액션 | 사람이 관리하는 해결 흐름 | 담당·기한·조치·검증·종료의 단일 기준이다. |
| 회의록 | 결정과 논의의 원문 | 문서 및 AI 분류 상태 | 이슈 발생 근거와 원문 앵커를 제공한다. |
| 주간업무 | 특정 주의 보고 내용 | 주차별 문서 상태 | 이슈 원장을 복제하지 않고 선택한 이슈의 스냅샷을 보고한다. |
| 대시보드 | 프로젝트 종합 현황 | 계산된 위험·진척 신호 | 이슈 후보를 제공하고 미해결 현황을 집계한다. |

### 2.1 지켜야 할 경계

- 이슈를 종료했다고 WBS 실적률을 자동으로 100%로 만들지 않는다.
- WBS 작업이 완료됐다고 연결된 이슈를 자동 종료하지 않는다.
- 회의록 AI 인사이트를 곧바로 이슈 테이블에 넣지 않는다.
- 주간업무 자유 텍스트와 이슈 원장을 양방향 자동 동기화하지 않는다.
- 대시보드 위험 신호가 사라졌다고 이미 등록된 이슈를 삭제하거나 종료하지 않는다.

## 4. 도메인 개념

### 4.1 이슈 유형

| 코드 | 표시명 | 정의 | 예시 |
|---|---|---|---|
| `issue` | 현안 | 이미 발생해 해결이 필요한 문제 | 인터페이스 테스트 오류, 기준정보 누락 |
| `risk` | 리스크 | 아직 발생하지 않았지만 대응이 필요한 위험 | UAT 일정 지연 가능성 |
| `action` | 액션 | 회의·검토에서 합의된 후속 조치 | 마스터 데이터 정합성 재검증 |

`decision` 인사이트는 완료된 의사결정 근거로 연결할 수 있지만, 기본 이슈 유형에는 넣지 않는다. 별도의 승인이 필요한 사안은 `issue` 또는 `action`으로 등록하고 제목·설명에 의사결정 필요 여부를 나타낸다.

### 4.2 상태

권장 상태 흐름:

```text
신규 ──→ 조치 중 ──→ 해결 요청 ──→ 종료
           │              │            │
           └──→ 보류 ─────┘            └──→ 재오픈 → 조치 중
```

| 코드 | 표시명 | 의미 | 주 변경 주체 |
|---|---|---|---|
| `new` | 신규 | 등록됐으나 담당·조치가 확정되지 않음 | 등록자, PMO |
| `in_progress` | 조치 중 | 담당자가 해결 작업을 수행 중 | 담당자, PMO |
| `on_hold` | 보류 | 외부 의존성·결정 대기 등으로 일시 정지 | 담당자, PMO |
| `resolution_requested` | 해결 요청 | 담당자가 조치 완료를 선언하고 검증 대기 | 담당자 |
| `closed` | 종료 | PMO가 해결 결과를 확인해 종료 | PMO |

`기한 초과`, `기한 임박`, `장기 미조치`는 상태가 아니라 날짜·활동 이력에서 계산하는 파생 플래그다.

### 4.3 우선순위

| 코드 | 표시명 | 의미 |
|---|---|---|
| `critical` | 긴급 | 즉시 대응하지 않으면 프로젝트 주요 일정·운영에 중대한 영향 |
| `high` | 높음 | 우선 대응이 필요하며 지연 시 영향이 큼 |
| `medium` | 보통 | 계획된 기한 안에 일반 절차로 처리 |
| `low` | 낮음 | 개선·관찰 성격이며 즉시 대응 필요성이 낮음 |

초기에는 영향도와 긴급도를 별도 필드로 나누지 않는다. 리스크 확률×영향 매트릭스가 필요해질 때 2차로 확장한다.

### 4.4 파생 상태

| 파생값 | 규칙 |
|---|---|
| 미해결 | `status != closed` |
| 기한 초과 | 미해결이고 `due_date < today` |
| 7일 내 마감 | 미해결이고 `today <= due_date <= today + 7일` |
| 장기 미조치 | 미해결이고 마지막 실질 업데이트 이후 설정 기간이 경과 |
| 내 담당 | 이슈 담당 멤버가 로그인 사용자의 프로젝트 멤버와 일치 |

## 5. 사용자와 권한

### 5.1 사용자 역할

| 사용자 | 주요 목적 |
|---|---|
| PMO 관리자 | 후보 검토, 담당·기한·우선순위 지정, 에스컬레이션, 해결 검증·종료 |
| 팀 편집자 | 자기 팀 이슈 등록, 담당 이슈 조치, 업데이트·증빙 제출, 해결 요청 |
| 일반 열람자 | 중요 이슈, 기한 초과, 원문 근거와 해결 경과 확인 |

### 5.2 권한 권장안

| 행위 | 일반 열람자 | 작성자 | 담당자·담당팀 | PMO |
|---|---:|---:|---:|---:|
| 목록·상세 조회 | O | O | O | O |
| 이슈 등록 | - | O | O | O |
| 댓글·조치 업데이트 | - | O | O | O |
| 제목·설명 수정 | - | O | O | O |
| 조치계획·해결결과 수정 | - | O | O | O |
| 담당·우선순위·기한 지정 | - | - | - | O |
| `해결 요청` 전환 | - | O | O | O |
| 종료·재오픈 | - | - | - | O |
| 보관 | - | - | - | O |
| 완전 삭제 | - | - | - | - |

현재 앱은 프로젝트 데이터 읽기를 인증 사용자 전체에 허용하는 관례가 있다. 민감 이슈가 예상된다면 이슈만 프로젝트 참여자에게 제한할지, 앱 전체 프로젝트 접근 모델을 바꿀지 구현 전에 결정해야 한다.

`memberships.role`은 인증 권한에 사용하고, `project_members.role`은 프로젝트 인력 명단 표시용 역할로 유지한다. 두 역할을 혼용하지 않는다.

위 표의 작성자·담당자·담당팀 사용자는 모두 `app_role()`이 있는 멤버십 사용자를 전제로 한다. 담당팀 사용자는 로그인 사용자의 `current_team()`이 이슈의 `owner_team_id`와 일치하는 경우다. 일반 열람자는 조회만 가능하며 업데이트·첨부를 작성할 수 없다.

## 6. 핵심 사용자 흐름

### 6.1 수동 등록

1. 사용자가 `+ 이슈 등록`을 누른다.
2. 빠른 등록에서 제목과 유형을 입력한다.
3. 필요하면 설명, 우선순위, 담당팀, 담당자, 기한, 연결 WBS를 지정한다.
4. 등록 후 이슈 상세 패널을 열어 조치계획과 첨부를 추가한다.
5. PMO가 미지정 정보를 보완하고 담당을 확정한다.

### 6.2 회의록 인사이트에서 등록

```text
회의록의 action/risk 인사이트
  → [이슈로 등록]
  → 제목·설명·원문 앵커 자동 채움
  → nearby deadline이 있으면 기한 후보로 채움
  → 사용자가 내용 확인·수정
  → 등록
```

- `action`, `risk`는 이슈 후보가 될 수 있다.
- `deadline`은 독립 이슈보다 연결된 후보의 기한 보조 정보로 사용한다.
- `decision`은 근거 링크로 연결하며 기본적으로 이슈를 자동 제안하지 않는다.
- 동일 회의록 블록·분류에서 이미 생성된 이슈가 있으면 새 등록 대신 기존 이슈 링크를 보여준다.

### 6.3 WBS 위험 신호에서 등록

1. 대시보드 위험 신호에서 `이슈로 등록`을 누른다.
2. 신호 제목·근거 수치·WBS 링크를 등록 폼에 채운다.
3. 사용자가 담당·기한·우선순위를 확인한다.
4. 등록 후 위험 신호는 기존 계산 결과로 계속 존재할 수 있으며, 연결 이슈 상태를 별도로 표시한다.

### 6.4 해결과 종료

1. 담당자가 조치 결과를 업데이트한다.
2. 증빙파일 또는 연결 산출물을 첨부한다.
3. 담당자가 해결 결과를 작성하고 `해결 요청`으로 전환한다.
4. PMO가 결과와 원문·증빙을 검토한다.
5. 적합하면 `종료`, 미흡하면 사유를 남기고 `조치 중`으로 되돌린다.
6. 종료 후 재발하면 PMO가 `재오픈`하며 기존 이력은 유지한다.

## 7. 정보 구조와 화면 설계

### 7.1 경로와 메뉴

- 프로젝트 메뉴: `/p/[projectId]/issues`
- 상세 딥링크 권장: `/p/[projectId]/issues?focus=[issueId]`
- 사이드바 위치: `칸반 보드` 다음
- 메뉴 라벨: `이슈관리`
- 메뉴 아이콘 후보: `CircleAlert`

`CircleAlert`를 채택하면 구현 시 `docs/design/icon-guideline.md`의 내비게이션 정본에 먼저 추가한다. 기존 `AlertTriangle`은 지연·위험 상태 의미로 유지해 메뉴 개념과 구분한다.

### 7.2 페이지 구성

```text
┌──────────────────────────────────────────────────────────────┐
│ ISSUE MANAGEMENT / 프로젝트 이슈관리             [+ 이슈 등록] │
│ [미해결 24] [긴급 3] [기한 초과 5] [해결 요청 2]              │
├──────────────────────────────────────────────────────────────┤
│ [검색] [내 담당] [상태] [우선순위] [팀] [출처]                 │
├──────────────────────────────────────────────────────────────┤
│ ISS-000123  인터페이스 오류      조치 중  긴급  ERP  D-2       │
│ ISS-000122  기준정보 검증 필요   신규     높음  MDM  기한초과   │
│ ...                                                           │
└──────────────────────────────────────────────────────────────┘
                                   선택 시 우측 상세 패널 →
```

현재 권장안에서는 `목록 / 보드` 뷰 전환을 Phase 1.2에서 이 툴바 앞에 추가하고, MVP 기본 화면에는 목록만 노출한다. §18에서 보드 출시 시점을 MVP로 확정하면 이 구성을 함께 앞당긴다.

### 7.3 상단 KPI

| KPI | 정의 | 이동 |
|---|---|---|
| 미해결 | 종료되지 않은 이슈 | 미해결 필터 적용 |
| 긴급 | 미해결 `critical` | 긴급 필터 적용 |
| 기한 초과 | 미해결이며 기한이 지남 | 기한 초과 필터 적용 |
| 해결 요청 | PMO 검증 대기 | 해결 요청 필터 적용 |

KPI는 이슈 레코드에서 결정적으로 계산한다. LLM으로 수치를 생성하지 않는다.

### 7.4 목록

권장 열:

1. 이슈 번호
2. 제목
3. 유형
4. 상태
5. 우선순위
6. 담당팀·담당자
7. 기한
8. 연결 WBS
9. 출처
10. 최근 변경

기본 정렬:

1. 미해결
2. 긴급 우선
3. 기한 초과 우선
4. 빠른 기한
5. 최근 변경

목록 행을 선택하면 우측 상세 패널을 열고 URL의 `focus`를 갱신한다. 새로고침·공유·챗봇 딥링크에서도 동일 이슈를 다시 열 수 있어야 한다.

### 7.5 상세 패널

상세 패널은 다음 순서로 구성한다.

1. 번호, 제목, 상태, 우선순위
2. 유형, 담당팀, 담당자, 기한
3. 설명과 영향
4. 조치계획과 해결 결과
5. 연결된 WBS·회의·회의록·주간업무
6. 첨부파일
7. 조치 업데이트
8. 상태·필드 변경 이력

PMO 전용 액션은 `종료`, `재오픈`, `보관`으로 분리한다. 종료·보관처럼 영향이 큰 액션은 확인 다이얼로그를 거친다.

### 7.6 등록·수정 폼

빠른 등록의 필수 항목은 제목과 유형으로 제한한다. 담당·기한이 없는 새 이슈는 `신규`로 등록하고 PMO의 분류 대기 큐에 노출한다.

상세 필드:

- 제목
- 유형
- 설명
- 우선순위
- 담당팀
- 담당자
- 기한
- 연결 WBS
- 발생 출처
- 조치계획

### 7.7 보드

보드를 MVP에 포함한다면 상태별 열을 사용한다.

- 신규
- 조치 중
- 보류
- 해결 요청
- 종료

보드 드래그는 권한이 있는 전환만 허용한다. 모바일과 키보드 사용자를 위해 드래그 외에 상태 선택 메뉴를 반드시 제공한다.

### 7.8 모바일

- KPI는 2열 그리드로 표시한다.
- 목록은 가로로 축소된 표가 아니라 카드 목록으로 전환한다.
- 필터는 접이식 패널 또는 전체 화면 필터로 제공한다.
- 상세 패널은 전체 화면 상세로 전환한다.
- 보드는 열 단위 가로 스크롤을 허용하되 상태 선택 메뉴를 기본 조작으로 제공한다.

## 8. 입력 규칙

| 필드 | 규칙 |
|---|---|
| 제목 | 필수, 앞뒤 공백 제거, 권장 최대 200자 |
| 설명 | 선택, 권장 최대 20,000자 |
| 유형 | `issue`, `risk`, `action` 중 하나 |
| 상태 | 허용된 상태 전환만 가능 |
| 우선순위 | `critical`, `high`, `medium`, `low` |
| 담당팀 | 기존 `teams` 참조 |
| 담당자 | 같은 프로젝트의 `project_members`만 가능 |
| 기한 | 날짜 형식 검증, 과거 날짜는 경고 후 허용 |
| 해결 결과 | `해결 요청` 전환 시 입력 권장, 종료 시 필수 |
| 연결 WBS | 같은 프로젝트의 WBS만 가능 |
| 출처 | 링크 대상과 프로젝트 일치 검증. 회의록은 연결된 회의의 프로젝트로만 판정 |
| 첨부 | 파일명, 크기, MIME, 허용 개수·용량 검증 |

등록 폼은 사용자가 과거 날짜의 이슈를 기록할 수 있도록 과거 기한을 완전히 차단하지 않는다. 대신 즉시 `기한 초과`로 표시될 수 있음을 안내한다.

현재 `minutes`에는 직접적인 `project_id`가 없고 `meeting_id`도 선택값이다. 따라서 MVP에서 프로젝트 이슈의 회의록 출처는 **프로젝트 회의에 연결된 회의록만 허용**한다. 연결되지 않은 전역 회의록은 먼저 같은 프로젝트의 회의와 연결해야 한다. 미연결 회의록까지 직접 연결해야 한다면 후속 마이그레이션으로 `minutes.project_id`를 도입하고 기존 데이터의 귀속·RLS 정책을 별도로 설계한다.

## 9. 데이터 모델

### 9.1 관계

```text
projects
   └── issues
         ├── issue_updates
         ├── issue_events
         ├── issue_attachments
         └── issue_links
                ├── wbs_items
                ├── meetings
                ├── minutes + 원문 블록 앵커
                └── weekly_report_rows
```

마이그레이션 번호는 구현 시점의 최신 번호 다음으로 정하고 롤백 파일을 함께 작성한다.

### 9.2 `issues`

| 컬럼 | 형식 | 설명 |
|---|---|---|
| `id` | uuid PK | 내부 식별자 |
| `issue_no` | bigint identity unique | 표시 번호의 원자적 발급 |
| `project_id` | uuid FK | 프로젝트 |
| `title` | text | 제목 |
| `description` | text | 설명·영향 |
| `type` | text check | `issue/risk/action` |
| `priority` | text check | `critical/high/medium/low` |
| `status` | text check | `new/in_progress/on_hold/resolution_requested/closed` |
| `owner_team_id` | uuid FK nullable | 담당팀 |
| `assignee_member_id` | uuid FK nullable | 프로젝트 멤버 담당자, 멤버 삭제 시 null |
| `reporter_user_id` | uuid FK nullable | 등록 행위자, 계정 삭제 시 null |
| `reporter_name` | text nullable | 탈퇴 후에도 표시할 이름 스냅샷 |
| `due_date` | date nullable | 완료 기한 |
| `action_plan` | text | 조치계획 |
| `resolution` | text | 해결 결과 |
| `version` | int | 낙관적 잠금 |
| `resolution_requested_at` | timestamptz nullable | 해결 요청 시각 |
| `closed_at` | timestamptz nullable | 종료 시각 |
| `archived_at` | timestamptz nullable | 보관 시각 |
| `created_at` | timestamptz | 생성 시각 |
| `updated_at` | timestamptz | 수정 시각 |

담당자는 프로젝트 인력인 `project_members`, 행위자는 인증 계정인 `auth.users`로 분리한다. 외부 인력이 프로젝트 멤버로 존재하더라도 로그인 계정이 없을 수 있기 때문이다.

`(assignee_member_id, project_id) → project_members(id, project_id)` 복합 FK로 다른 프로젝트 멤버가 담당자로 섞이는 것을 DB에서 차단한다. 저장소의 근태 무결성 마이그레이션과 같은 방식이다. 멤버 삭제 처리는 주의가 필요하다: 무지정 `ON DELETE SET NULL`은 참조 컬럼 전체(NOT NULL인 `project_id` 포함)를 null 처리 대상으로 삼아 멤버 삭제가 런타임 실패한다. PG15+ 컬럼 지정 구문 `on delete set null (assignee_member_id)`을 사용한다 — 프로덕션 PostgreSQL 17.6 실측 확인(2026-07-23)으로 트리거 대안은 불필요. 근태 0032가 만든 유니크 인덱스 `project_members_id_project_uidx`를 재사용하고, issues는 신규 테이블이라 NOT VALID 단계 없이 즉시 생성한다.

사용자·멤버 삭제가 이슈 이력을 막지 않도록 다음 삭제 동작을 명시한다.

- `reporter_user_id`, 업데이트·이벤트의 actor/author, 첨부의 `created_by`: `ON DELETE SET NULL`
- `assignee_member_id`: 멤버 삭제 시 담당자만 null 처리
- 이름 스냅샷(`reporter_name`, `author_name`, `actor_name`)은 그대로 보존

### 9.3 `issue_updates`

| 컬럼 | 설명 |
|---|---|
| `id`, `issue_id` | 업데이트 식별자와 부모 |
| `kind` | `comment`, `progress`, `resolution`, `rejection` |
| `body` | 업데이트 내용 |
| `author_user_id`, `author_name` | 작성자와 이름 스냅샷, 계정 삭제 시 ID만 null |
| `created_at`, `edited_at` | 생성·수정 시각 |
| `deleted_at` | 완전 삭제 대신 내용 숨김 시각 |

자유 댓글과 조치 업데이트를 같은 테이블에서 관리하되 `kind`로 의미를 구분한다.

### 9.4 `issue_events`

상태 전환과 주요 필드 변경을 기록하는 append-only 감사 이력이다.

| 컬럼 | 설명 |
|---|---|
| `id`, `issue_id` | 이벤트 식별자와 부모 |
| `event_type` | `created`, `updated`, `assigned`, `status_changed`, `closed`, `reopened`, `archived` 등 |
| `actor_user_id`, `actor_name` | 행위자와 이름 스냅샷, 계정 삭제 시 ID만 null |
| `old_values` | 변경 전 주요 값 JSON |
| `new_values` | 변경 후 주요 값 JSON |
| `created_at` | 발생 시각 |

기존 `change_logs`는 WBS 전용 구조이므로 확장하지 않고 이슈 전용 이벤트 테이블을 둔다.

### 9.5 `issue_links`

| 컬럼 | 설명 |
|---|---|
| `id`, `issue_id` | 링크 식별자와 부모 |
| `link_type` | `wbs_item`, `meeting`, `minute_block`, `weekly_row`, `risk_signal` |
| `wbs_item_id` | WBS 링크 |
| `meeting_id` | 회의 링크 |
| `minute_id` | 회의록 링크 |
| `weekly_row_id` | 주간업무 행 링크 |
| `block_index`, `block_hash`, `body_hash` | 회의록 원문 앵커 |
| `source_key` | 중복 등록 방지용 결정적 키 |
| `source_project_id` | 링크 생성 시 검증한 프로젝트 귀속 스냅샷 |
| `label` | 표시용 출처 라벨 |
| `target_snapshot` | 원본 ID·제목·경로 등 삭제 후에도 남길 최소 스냅샷 |
| `link_state` | `active`, `stale`, `missing` |
| `broken_at` | 원본 삭제 등으로 실시간 링크가 끊긴 시각 |
| `metadata` | 신호 수치 등 비핵심 부가정보 JSON |
| `created_at` | 연결 시각 |

다형 문자열 ID 하나에 모든 대상을 넣지 않는다. 각 대상의 nullable FK를 둔다. 활성 링크는 `link_type`에 맞는 대상 FK 하나만 채워져야 한다. `missing` 링크는 대상 FK가 모두 null이어야 하고, `stale` 링크는 원본은 존재하지만 본문 앵커나 프로젝트 귀속이 달라진 경우라 대상 FK를 유지할 수 있다. `stale/missing`은 모두 `broken_at`과 `target_snapshot`이 있어야 한다는 CHECK 제약을 둔다.

대상 FK는 `ON DELETE SET NULL`을 사용하고 DB 트리거가 `link_state='missing'`, `broken_at`을 기록한다. `target_snapshot`은 링크 생성 시 항상 저장한다. 따라서 원본이 삭제돼도 이슈 타임라인에는 당시 출처 제목과 ID가 `원본 없음` 상태로 남는다.

회의록 링크를 만들 때는 `minutes.meeting_id → meetings.project_id`를 검증한 뒤 그 프로젝트를 `source_project_id`와 `target_snapshot`에 박제한다. 이후 회의가 삭제되거나 회의록이 다른 회의로 재연결돼 귀속을 다시 검증할 수 없거나 달라지면 minute FK 자체는 유지하되 링크를 `stale`로 바꾼다. 이미 검증된 과거 근거는 보존하지만 새로운 활성 프로젝트 링크로 간주하지 않는다.

`source_key` 예:

```text
minute:{minuteId}:{blockIndex}:{blockHash}:{kind}
risk-signal:{projectId}:{signal.id}
weekly:{weeklyRowId}:{field}:{contentHash}
```

원문에서 이슈를 생성할 때 같은 `source_key`가 이미 있으면 중복 등록을 막고 기존 이슈를 안내한다.

위험 신호 키는 전체 위험 보고서 fingerprint도, 근거(evidence) 목록 기반 키도 사용하지 않는다. 근거 목록은 `EVIDENCE_LIMIT`(8건) 절단과 항목 완료·순위 변동으로 상시 바뀌어 같은 개념의 신호가 시간에 따라 다른 키를 갖게 되므로 중복 방지·재오픈 제안이 무력화된다(2026-07-23 코드 검증). 대신 신호 엔티티의 안정 id를 그대로 쓴다: `delay_trend`, `deadline_stall`, `owner_overload:{팀}`, `overdue_accumulation`, `meeting_action_stale` (`src/lib/domain/riskSignals.ts`) — 5종 모두 프로젝트 내 단일 발화라 이것으로 결정적이다. 건수·SPI·근거 목록 같은 변동 수치는 `metadata` 스냅샷에 저장한다. 같은 신호가 종료 후 재발하면 새 이슈를 자동 생성하지 않고 기존 종료 이슈의 `재오픈`을 제안한다.

### 9.6 `issue_attachments`

| 컬럼 | 설명 |
|---|---|
| `id`, `issue_id` | 첨부 식별자와 부모 |
| `update_id` | 특정 조치 업데이트에 붙은 경우 참조 |
| `file_name`, `file_path` | 표시명과 Storage 경로 |
| `size`, `mime` | 파일 메타데이터 |
| `created_by`, `created_at` | 업로더와 시각 |
| `deleted_at` | 보관 또는 삭제 표시 |

Storage 경로 권장:

```text
{projectId}/{issueId}/{randomUuid}/{safeFileName}
```

버킷은 비공개로 두고 Storage RLS도 이슈 접근 권한과 일치시킨다.

### 9.7 2차 테이블

- `issue_watchers`: 구독 사용자
- `issue_notifications`: 사용자별 영속 알림, `read_at`, `dedupe_key`

현재 WBS 알림의 환경설정 JSON 방식은 지속적인 배정·댓글·상태 알림에 적합하지 않으므로 이슈 알림은 별도 행으로 관리한다.

## 11. 보안과 RLS

### 11.1 기본 원칙

- 서버 액션 권한 선검증과 Postgres RLS를 함께 적용한다.
- 서버 액션 성공 여부만 믿지 않고 UPDATE·DELETE 대상 행이 실제로 반환됐는지 확인한다.
- 클라이언트가 보낸 project ID, 담당자, 출처 ID를 그대로 신뢰하지 않는다.
- 담당자와 링크 대상이 이슈와 같은 프로젝트인지 DB와 서버 양쪽에서 확인한다. 단, 회의록은 `minutes.meeting_id → meetings.project_id`로 귀속을 검증하며 미연결 회의록은 MVP에서 거부한다.
- hard delete 정책을 만들지 않는다.
- 감사 이벤트는 일반 사용자가 수정·삭제할 수 없다.

### 11.2 읽기 권한

MVP 권장안은 현재 앱의 관례와 동일하게 인증 사용자 전체 읽기다. 다만 다음과 같은 내용이 들어올 수 있다면 프로젝트 참여자 제한이 필요하다.

- 공급업체 계약·책임 소재
- 인사·조직 관련 문제
- 보안 취약점
- 원가·손실 정보

이 경우 이슈만 좁힐지, 프로젝트 전체 읽기 정책을 재설계할지 별도 결정한다.

### 11.3 쓰기 권한

- 생성: `app_role()`이 있는 로그인 사용자
- 제목·설명 수정: 작성자, 담당자, 담당팀 사용자. 종료·보관 상태에서는 PMO만
- 조치계획·해결결과 수정: 작성자, 담당자, 담당팀 사용자
- `new/in_progress/on_hold/resolution_requested` 안의 허용 상태 전환: 작성자, 담당자, 담당팀 사용자
- 담당팀·담당자·우선순위·기한 변경: PMO
- 종료·재오픈·보관: PMO
- 업데이트 작성: `app_role()`이 있는 작성자, 담당자, 담당팀 사용자와 PMO
- 첨부: 이슈 수정 또는 업데이트 작성 권한 보유자
- 감사 이벤트: DB 함수·트리거만 insert

비 PMO의 수정 가능 컬럼을 DB에서도 제한해야 한다면 전용 RPC 또는 OLD/NEW 비교 트리거를 사용한다. RLS에서 단순히 행 UPDATE만 열면 사용자가 담당·우선순위까지 임의로 바꿀 수 있다.

## 12. 기존 기능 연동

### 12.1 WBS

- 이슈에서 관련 WBS 행을 하나 이상 연결할 수 있다.
- 링크는 `/p/[projectId]/wbs?focus=[itemId]`로 이동한다.
- WBS 상세 패널에는 2차로 미해결 이슈 배지를 표시한다.
- 이슈 상태와 WBS 실적률은 자동 동기화하지 않는다.

### 12.2 회의록

- `action`, `risk` 인사이트에서 이슈 등록 후보를 만든다.
- 등록 폼에 회의록 제목·일자·원문 블록 링크를 채운다.
- MVP에서는 `meeting_id`가 있고 그 회의가 현재 프로젝트에 속한 회의록만 링크할 수 있다.
- 미연결 전역 회의록은 먼저 프로젝트 회의와 연결해야 하며, 연결할 수 없으면 원문 링크 없이 수동 이슈로 등록한다.
- 원문 이동은 기존 블록 앵커와 body hash 계약을 재사용한다.
- 동일 원문 source key 중복을 차단한다.
- 회의록 본문의 body hash가 달라지면 자동 재매칭하지 않는다. 링크를 `stale`로 표시하고 당시 `target_snapshot`을 보여주며, 문서 상단으로만 이동한다.
- 사용자가 현재 원문 블록을 확인해 수동으로 다시 연결하면 새 block index/hash/body hash로 갱신하고 그 행위를 `issue_events`에 기록한다.
- 회의 삭제나 회의록의 meeting 연결 변경으로 프로젝트 귀속이 사라지거나 달라지면 링크를 `stale`로 표시하고 생성 당시 `source_project_id` 스냅샷을 보존한다.

### 12.3 회의

- 이슈와 관련 회의를 링크할 수 있다.
- 회의 상세에서 관련 이슈 목록을 보여주는 기능은 2차로 둔다.
- 회의 참석자를 이슈 담당자로 자동 지정하지 않는다.

### 12.4 주간업무

- 이슈 원장을 주간업무 자유 텍스트와 자동 동기화하지 않는다.
- 사용자가 선택한 이슈를 해당 주차 보고에 스냅샷으로 삽입한다.
- 삽입 이후 원본 이슈가 바뀌어도 이미 제출된 과거 보고 문장을 소급 변경하지 않는다.

### 12.5 대시보드

2차 연동:

- 미해결
- 긴급
- 기한 초과
- 해결 요청
- 이번 주 신규·종료
- 팀별 미해결

기존 규칙 기반 위험 신호는 이슈 후보를 제공한다. 신호가 사라져도 사람이 등록한 이슈는 자동 종료하지 않는다.

### 12.6 알림

알림 대상 권장:

- 담당자 배정
- 담당자 또는 기한 변경
- 기한 임박
- 기한 초과
- 해결 요청
- PMO의 종료 거절
- 종료·재오픈

댓글마다 모든 사람에게 알리는 방식은 피한다. 구독자 또는 직접 관련자에게만 보내고 같은 이벤트는 `dedupe_key`로 중복을 막는다.

### 12.7 DK Bot

MVP 이후 다음 질의를 지원할 수 있다.

- 내 미해결 이슈
- 이번 주 기한 초과 이슈
- ERP 팀 긴급 이슈
- 특정 WBS에 연결된 이슈
- 최근 종료된 이슈와 해결 결과

Bot이 상태를 바꿀 때는 기존 실행 봇과 마찬가지로 변경안을 보여주고 사용자 확인 후 실행한다.

## 13. 예외와 실패 처리

| 상황 | 처리 |
|---|---|
| 동일 원문에서 중복 등록 | 신규 생성 대신 기존 이슈 링크 표시 |
| 연결된 WBS·회의·회의록 삭제 | FK는 null 처리하고 `target_snapshot`·`broken_at`을 보존해 `원본 없음`으로 표시 |
| 회의록 본문 또는 프로젝트 귀속 변경 | 링크를 `stale`로 전환하고 스냅샷을 표시, 사용자 확인 후 수동 재연결 |
| 담당 프로젝트 멤버 삭제 | 담당자를 null 처리하고 재배정 필요 배지 표시 |
| 담당자에게 로그인 계정이 없음 | 표시·배정은 허용하되 개인 알림은 보내지 못함을 안내 |
| 상태 저장 중 동시 수정 | version 충돌 안내 후 최신 데이터를 다시 불러옴 |
| 해결 결과 없이 종료 | 서버에서 거부 |
| 권한 없는 상태 전환 | UI 버튼 미노출 + 서버/RLS에서 재차 거부 |
| 첨부 메타 저장 실패 | Storage 업로드 파일을 보상 삭제하거나 고아 파일 정리 대상으로 기록 |
| 이슈 보관 | 기본 목록에서 숨기고 감사·검색 기록은 유지 |
| AI 후보 생성 실패 | 수동 등록은 정상 제공하고 AI 실패를 빈 후보로 위장하지 않음 |

## 14. 비기능 요구사항

### 14.1 감사 가능성

- 주요 필드 변경 전·후를 남긴다.
- 상태 전환, 담당 변경, 기한 변경, 종료·재오픈·보관을 반드시 기록한다.
- 사용자 탈퇴 후에도 이름 스냅샷으로 행위자를 표시한다.
- 이벤트 이력은 일반 사용자가 수정하거나 삭제할 수 없다.

### 14.2 성능

- 목록 조회는 설명 전문·첨부 전문을 포함하지 않는다.
- 기본 페이지 크기 또는 커서 페이지네이션을 사용한다.
- 인덱스 후보:
  - `(project_id, status, updated_at desc)`
  - `(project_id, assignee_member_id, status)`
  - `(project_id, due_date) where status <> 'closed'`
  - `(project_id, priority, status)`
  - `issue_links(source_key)` unique where source_key is not null
- KPI와 목록 데이터를 가능하면 같은 조회 왕복에서 조립하거나 병렬 조회한다.

### 14.3 접근성

- 상태·우선순위는 색상만으로 구분하지 않고 텍스트를 함께 표시한다.
- 보드의 모든 드래그 동작에 버튼·메뉴 대안을 제공한다.
- 아이콘 전용 버튼에 `aria-label`을 제공한다.
- 상세 패널의 열기·닫기와 모달의 포커스 복귀를 보장한다.
- 오류와 저장 상태는 `aria-live` 또는 Toast로 전달한다.

### 14.4 반응형

- 데스크톱은 목록 + 우측 상세 패널
- 모바일은 카드 목록 + 전체 화면 상세
- 긴 필터 툴바는 줄바꿈 또는 필터 패널로 전환
- 표의 필수 정보는 제목·상태·담당·기한이며 나머지는 모바일 상세로 이동

### 14.5 다국어

새 라벨은 `src/lib/i18n/dict/issues.ts`로 분리하고 한국어·영어 키를 동시에 추가한다. DB에는 번역된 표시문 대신 안정적인 영문 코드를 저장한다.

- 입력 검증

- 역할별 권한
- 기한 초과·임박 계산
- 기본 정렬
- source key 결정성
- 같은 프로젝트 담당자 검증

### 15.2 서버 액션 테스트

- 인증·권한 게이트
- 생성·수정·상태 전환
- version 충돌
- 0행 UPDATE 무음 성공 방지
- 상태 변경과 이벤트 기록의 원자성
- 같은 source key 중복 등록
- 보관·복원
- 다른 프로젝트 담당자·링크 혼입 차단

### 15.3 UI 테스트

- 필터와 검색
- URL `focus`에 따른 상세 열기
- 역할별 버튼 노출
- 해결 요청·종료 확인 흐름
- 오류·로딩·빈 상태
- 모바일 카드 렌더
- 키보드 상태 변경

### 15.4 DB·RLS 검증

- 익명 접근 거부
- 인증 사용자 읽기 정책
- 일반 사용자와 PMO 쓰기 차이
- 감사 이벤트 수정·삭제 거부
- 다른 프로젝트 멤버 담당 지정 거부
- 비공개 첨부 Storage 접근

### 15.5 통합 시나리오

1. 회의록 action 인사이트에서 이슈 후보를 연다.
2. 담당자·기한을 확인하고 등록한다.
3. 동일 원문에서 다시 등록을 시도해 중복이 차단되는지 확인한다.
4. 담당자가 조치 업데이트와 첨부를 남긴다.
5. 담당자가 해결 요청한다.
6. PMO가 종료한다.
7. 대시보드·필터에서 종료 결과가 반영되는지 확인한다.
8. 재오픈 후 전체 감사 이력이 유지되는지 확인한다.

## 16. 단계별 출시

현재 권장안 기준으로 MVP 완료 범위는 Phase 1과 Phase 1.1을 합친 것이며 Phase 1.2는 MVP 이후 운영 편의 기능이다. §18에서 보드·첨부 출시 시점을 다르게 확정하면 단계 경계를 함께 조정한다.

### Phase 1 — 핵심 이슈 원장

- DB·RLS·도메인 모델
- 프로젝트 메뉴
- 목록·KPI·검색·필터
- 등록·수정·상세
- 담당·기한·상태 전환
- 조치 업데이트
- 변경 이력
- 보관

### Phase 1.1 — 근거와 증빙

- WBS·회의·회의록 링크
- 회의록 인사이트에서 확인 후 등록
- 위험 신호에서 확인 후 등록
- source key 중복 방지
- 첨부파일

### Phase 1.2 — 운영 편의

- 상태별 보드

- 영속 알림

- WBS·칸반 미해결 이슈 배지

- 글로벌 내 이슈
- DK Bot 이슈 질의와 확인형 변경

## 17. MVP 수용 기준

- [ ] 프로젝트 사이드바에서 이슈관리 메뉴에 진입할 수 있다.
- [ ] 사용자가 제목과 유형으로 이슈를 등록할 수 있다.
- [ ] 이슈 번호가 동시 등록에서도 중복 없이 발급된다.
- [ ] PMO가 우선순위, 담당팀, 담당자, 기한을 지정할 수 있다.
- [ ] 담당자가 조치 업데이트와 해결 결과를 작성할 수 있다.
- [ ] 권한에 맞는 상태 전환만 가능하다.
- [ ] PMO만 종료·재오픈·보관할 수 있다.
- [ ] 미해결, 긴급, 기한 초과, 해결 요청 KPI가 정확하다.
- [ ] 검색과 상태·우선순위·팀·담당자·기한 필터가 동작한다.
- [ ] 목록 행 또는 딥링크로 같은 상세 이슈를 열 수 있다.
- [ ] 연결된 WBS와 회의록 원문으로 이동할 수 있다.
- [ ] 같은 원문 source key의 중복 이슈 등록이 차단된다.
- [ ] 주요 변경과 상태 전환이 감사 이력에 남는다.
- [ ] 이슈를 보관해도 이력과 연결 정보가 유지된다.
- [ ] 모바일에서 핵심 목록과 상세를 사용할 수 있다.
- [ ] AI·위험 신호가 사용자 확인 없이 이슈를 생성하지 않는다.

## 18. 구현 전 결정할 사항

| 항목 | 선택지 | 권장안 |
|---|---|---|
| 관리 범위 | 현안만 / 현안+리스크+액션 | 통합 실행 큐 |
| 상태 | 4단계 / 보류 포함 5단계 | `보류` 포함 |
| 종료 권한 | 담당자 종료 / PMO 확인 종료 | `해결 요청 → PMO 종료` |
| 기본 화면 | 목록 / 보드 / 상황판 | 목록 |
| 보드 출시 | MVP / 후속 | Phase 1.2 |
| 읽기 범위 | 인증 사용자 전체 / 프로젝트 참여자 | 현행 유지, 민감성 확인 후 재결정 |
| 이슈 번호 | 전역 / 프로젝트별 | 전역 identity `ISS-000123` |
| 업데이트 | 자유 댓글 / 구조화 조치 기록 | 한 테이블 + kind 구분 |
| 첨부 | MVP / 후속 | Phase 1.1 |
| AI 연동 | 자동 등록 / 후보 확인 | 후보 확인 |
| 주간업무 연동 | 자동 동기화 / 선택 삽입 | 선택 스냅샷 삽입 |

**2026-07-23 확정 결과 (정본: [MVP 스펙](../superpowers/specs/2026-07-23-issues-mvp-design.md)):** 관리 범위=단일 유형(현안) · 상태=4단계(보류 포함, 해결요청 없음) · 종료=멤버 직접 해결(회의 패턴) · 기본 화면=목록+모달 · 보드=Phase 2+ · 읽기=현행 유지 · 번호=전역 identity(`#12` 표기, `ISS-000123` 포맷 불채택) · 업데이트=단일 조치 메모 컬럼(테이블은 Phase 2+) · 첨부=Phase 2+ · AI 연동=후보 확인 방식으로 Phase 2+ · 주간업무 연동=선택 스냅샷을 시트 셀 텍스트로 삽입(주간 PPT 파이프라인은 시트 셀만 읽으므로 PPT 개조 0 — 이슈 테이블을 PPT가 직접 읽는 개조로 흐르지 않도록 명시).

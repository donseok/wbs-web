# sub-act 중복 표시 정리 — 트리 축약 표시 + 기본 접힘

2026-07-03 · 대상: WBS 시트(`WbsGanttSheet`)

## 문제

복수 담당 말단 act 는 임포트 시 담당별 sub-act 로 분리된다(팀별 실적 입력을 위해 유지).
sub-act 의 DB 이름은 `"{부모 작업명} ({팀} 주관/지원)"` — 리프 이름만 소비하는 하류(DK Bot
검색·색인, 주간보고, 알림, 대시보드, 칸반)에서 작업 식별이 가능해야 하기 때문이다.
그런데 트리 화면에서는 부모가 바로 위 행에 보이므로 이 접두가 순수 중복으로 보이고,
이름 칸(300px)에서 잘리면 부모와 자식 4행이 전부 같은 텍스트로 보인다.

## 결정

**저장 이름은 그대로 두고, 트리에서만 축약 표시 + 복수 담당 부모는 기본 접힘.**

1. **표시명 축약** — 부모가 activity 인 항목(sub-act)은 이름 칸에 `└ 가공 주관` 처럼
   팀 부분만 표시한다.
   - 규칙: `name` 이 부모 `name` 으로 시작하면 접두 제거 후 바깥 괄호 한 겹을 벗긴다.
     접두가 일치하지 않으면(사용자가 개명한 경우) 전체 이름을 그대로 표시한다.
   - 스타일: 이름은 `text-ink-muted`, 접속선 `└` 은 `text-ink-subtle`. hover title 은
     풀네임 유지(기존 title 그대로).
   - 상세 패널·검색(buildMatch)·챗봇·보고서·알림은 풀네임을 계속 쓴다 — 변경 없음.
2. **기본 접힘** — `collapsed` 초기값을 "activity 이면서 자식이 있는 항목(id)" 집합으로
   설정한다. 첫 화면이 엑셀 원본과 같은 67행이 되고, 부모 act 가 담당 배지(●/△)를
   보유하므로 접힌 상태에서 정보 손실이 없다. 실적 입력 시 ▸ 로 펼친다.
   - "전체 펼치기/접기" 토글, 검색 시 전체 펼침(기존 동작), 새로고침 후 상태 유지
     (클라이언트 state)는 기존 메커니즘 그대로.

## 고려한 대안

- **축약 표시만(항상 펼침)**: 행 수 168 유지 — 입력은 빠르나 첫 화면이 여전히 번잡.
- **DB 이름 자체 축약 + 하류 6곳 부모명 결합**: 데이터는 깔끔하나 수정 면적·회귀 위험이
  가장 큼(직전 리뷰에서 이 경로의 누락 위험이 실제 지적됨). 기각.

## 구현 계획

1. `WbsGanttSheet`: `subActIds`(Set) → `subActLabels`(Map<id, 축약명>)로 확장.
   이름 버튼 렌더에서 sub-act 면 `└ {축약명}` + muted 스타일. `collapsed` 초기값을
   분리 부모 id 집합으로. SUB-ACT 배지는 기존 유지.
2. jsdom 테스트: 축약 표시(접두 제거·개명 폴백)·기본 접힘·펼침 후 sub-act 노출.
3. tsc/lint/vitest/build → 커밋 → 배포.

## 테스트

- 분리 부모는 기본 접힘, 첫 flatRows 에 sub-act 미포함.
- 펼치면 sub-act 가 `└ 가공 주관` 축약명으로 보인다.
- 부모명 접두가 없는(개명된) sub-act 는 풀네임으로 보인다.
- 검색 시(전체 펼침) 축약명 표시 유지, 매칭은 풀네임 기준.

# 주간보고 점검 기능 설계

작성일: 2026-07-22

## 배경

주간업무 작성 화면(`/p/[projectId]/weekly`)은 업무영역 구분 10행 × 내용 4열(금주실적 내용/이슈, 차주계획 내용/이슈)의 시트다. 각 셀은 Alt+Enter로 여러 줄을 담는다.

작성자가 여러 명이고 이월(carry-over)로 전주 내용이 자동 유입되기 때문에, 서로 다른 구분에 같은 문장이 중복으로 남거나, 셀 안 줄 번호가 어긋나거나, 글머리 기호·공백이 제각각인 상태로 PPT 내보내기까지 가는 일이 생긴다.

작성 화면에서 이 문제들을 찾아 보여주고, 항목별로 수정을 적용할 수 있게 한다.

## 범위

- 점검 대상: **현재 보고 있는 주차의 내용 4개 열만**. 구분·모듈 열은 편집 대상이 아니므로 제외한다.
- 판정 방식: **코드 규칙(문자열/정규식)만**. LLM은 쓰지 않는다. 결과가 결정적이어야 "적용" 버튼이 안전하다.
- 수정 방식: **항목별 개별 적용**. 일괄 자동 수정은 넣지 않는다.

## 점검 규칙

### ① 같은 열, 구분 간 중복

4개 열 각각에 대해 10개 구분 행의 **줄 단위**로 비교한다.

비교 전 정규화(비교 전용 — 저장 값에는 영향 없음):

1. 앞뒤 공백 제거
2. 선두 글머리 기호(`-` `·` `*` `●`) 제거
3. 선두 줄 번호(`숫자` + `.` 또는 `)`) 제거
4. 연속 공백을 1칸으로 축약

즉 `- 설계 리뷰 완료`와 `1. 설계  리뷰 완료`는 같은 줄로 본다. 정규화 결과가 빈 문자열인 줄은 비교에서 제외한다.

- **판정**: 정규화 결과가 같은 줄이 서로 다른 구분 행에 존재하면 지적.
- **표시**: `금주실적 내용 — 영업 · 구매에 같은 줄이 있습니다: "설계 리뷰 완료"`
- **수정**: `sortOrder`가 가장 작은 구분 행의 줄만 남기고, 나머지 구분 행에서 해당 줄을 삭제한다. 3개 이상 겹쳐도 지적 1건으로 묶고, 한 번의 적용으로 나머지 전부를 지운다.
- 같은 셀 안의 중복은 이 규칙의 대상이 아니다(검사하지 않는다).

### ② 셀 안 줄 번호 체번

한 셀 안에서 `숫자` + `.` 또는 `)` 로 시작하는 줄이 **2줄 이상**일 때만 검사한다. 번호를 쓰지 않는 셀은 지적하지 않는다.

- **판정**: 번호가 1부터 1씩 증가하지 않으면 지적. 건너뜀(1,2,4), 중복(1,2,2), 시작값 오류(2,3,4), 역순 모두 포함.
- **수정**: 첫 번째 번호 줄의 스타일 — 구분자가 `.` 인지 `)` 인지, 구분자 뒤 공백이 있는지 — 를 그대로 따라 1부터 다시 매긴다. 번호 없는 줄은 순서·내용을 그대로 두고 번호도 소비하지 않는다.
- **표시**: `PMO · 금주실적 내용 — 줄 번호가 1, 2, 4 입니다 → 1, 2, 3`

### ③ 공백·빈줄·기호 정리

셀 하나당 지적 1건으로 묶는다(줄마다 쪼개면 목록이 폭발한다).

- 줄 끝 공백 제거
- 줄 안 연속 공백 → 1칸
- 전각 공백(U+3000) → 반각 공백
- 셀 맨 앞/뒤 빈 줄 제거
- 셀 중간 연속 빈 줄 2줄 이상 → 1줄
- 글머리 기호 통일: **시트 전체에서 가장 많이 쓰인 기호**로 맞춘다. 동수면 `-` 우선. 시트 전체에 기호가 한 종류만 쓰였으면 지적하지 않는다.

정리 결과가 원본과 동일하면 지적하지 않는다.

- **표시**: `구매 · 차주계획 내용 — 줄 끝 공백 2곳, 빈 줄 1곳, 글머리 기호 · → -`

## 화면 동작

**버튼**: 상단 고정 툴바(`WeekNav`)의 PPT 내보내기 버튼 왼쪽에 `주간보고 점검`. 내보내기 전에 점검하는 순서가 자연스럽다.

**패널**: 기존 `src/components/ui/Modal.tsx`를 쓴다. 여는 순간 현재 화면의 `rows`(미저장 dirty 셀 포함)로 점검을 돌린다. 서버 왕복 없음.

```
┌─ 주간보고 점검 ───────────────────── 3건 ─┐
│ ⚠ 중복  금주실적 내용                      │
│    영업 · 구매에 같은 줄                   │
│    "설계 리뷰 완료"                        │
│    → 구매에서 이 줄을 지웁니다   [적용]    │
│ ─────────────────────────────────────────  │
│ ⚠ 체번  PMO · 금주실적 내용                │
│    줄 번호 1, 2, 4 → 1, 2, 3     [적용]   │
│ ─────────────────────────────────────────  │
│ ℹ 정리  구매 · 차주계획 내용               │
│    줄 끝 공백 2곳, 글머리 · → -   [적용]   │
└────────────────────── [닫기] ──────────────┘
```

- 항목의 **제목 부분 클릭** → 모달이 닫히고 해당 셀로 선택이 이동한다. 어디를 말하는지 눈으로 확인하고 직접 고칠 수 있게.
- **[적용]** → `runBatch(edits, { undoable: true })`로 해당 셀만 저장. 모달은 열린 채로 유지되고, 목록은 갱신된 `rows`로 즉시 다시 계산된다. 방금 고친 항목은 사라지고, 그 수정 때문에 새로 드러난 항목(중복 줄을 지웠더니 번호가 어긋남 등)이 바로 보인다.
- 되돌리기: 적용 후 모달을 닫고 `Ctrl+Z`. 기존 undo 스택을 그대로 타므로 별도 구현이 없다.
- 지적이 없으면 `점검할 내용이 없습니다.` 빈 상태를 표시한다.
- 저장 실패는 기존 `runBatch`의 재시도 칩이 그대로 처리한다. 패널은 저장 성공/실패를 따로 추적하지 않는다.

## 구조

| 파일 | 역할 | 상태 |
|---|---|---|
| `src/lib/domain/weeklyLint.ts` | 점검 규칙(순수, I/O 없음). `lintWeeklySheet(rows) → LintFinding[]` | 신규 |
| `tests/domain/weeklyLint.test.ts` | 규칙별 단위 테스트 | 신규 |
| `src/components/weekly/WeeklyLintPanel.tsx` | 모달 UI + 적용 배선 | 신규 |
| `src/components/weekly/useSheetGrid.ts` | `focusCell(addr)` 노출 | 수정(소) |
| `src/components/weekly/WeeklySheetView.tsx` | 버튼·모달 렌더 및 `rows`/`runBatch` 전달 | 수정(소) |

`WeeklySheetView.tsx`가 이미 754줄이므로 패널을 별도 파일로 두어 더 키우지 않는다.

### 인터페이스

```ts
export type LintKind = 'duplicate' | 'numbering' | 'format'

export interface LintFinding {
  /** 안정 키(React list). `${kind}:${rowId}:${cellKey}` 를 기본으로, 중복은 정규화 줄 해시를 덧붙인다. */
  id: string
  kind: LintKind
  /** 클릭 시 이동할 대표 셀. 중복은 '삭제 대상' 중 sortOrder가 가장 작은 행(= 실제로 바뀌는 첫 셀),
   *  체번·정리는 그 셀 자신. */
  rowId: string
  cellKey: WeeklyCellKey
  /** 목록 제목 — 예: `PMO · 금주실적 내용` */
  title: string
  /** 무엇이 문제이고 적용하면 어떻게 되는지 */
  detail: string
  /** 적용할 편집. 기존 배치 편집 단위를 그대로 쓴다. */
  edits: WeeklyCellEdit[]
}

export function lintWeeklySheet(rows: WeeklySheetRow[]): LintFinding[]
```

`edits`가 기존 `WeeklyCellEdit[]` 타입이므로 패널은 새 저장 개념을 만들지 않는다. 이미 있는 `runBatch`에 그대로 넘긴다.

정렬 순서: `duplicate` → `numbering` → `format`, 각 부류 안에서는 열 순서(`this_content` → `this_issue` → `next_content` → `next_issue`) 다음 행 `sortOrder`.

### 데이터 흐름

```
rows (WeeklySheetView state, dirty 셀 포함)
  → lintWeeklySheet(rows)            // 순수, useMemo
  → LintFinding[]
  → WeeklyLintPanel 렌더
  → [적용] → runBatch(finding.edits, { undoable: true })
  → rows 갱신 → lintWeeklySheet 재실행 → 목록 갱신
```

`lintWeeklySheet`는 모달이 열려 있는 동안 `rows`가 바뀔 때마다 재실행된다(10행 × 4열이므로 비용이 무시할 만하다). Realtime으로 타인의 수정이 들어와도 목록이 자동으로 따라간다.

## 오류 처리

- 점검은 순수 계산이라 실패 경로가 없다. 예외를 던지지 않는다.
- 적용 시점에 대상 행이 사라졌거나(Realtime DELETE) 셀 값이 지적 당시와 달라졌으면, `runBatch`가 계산된 `edits`를 그대로 덮어쓴다. 이는 `rows` 변경 시 목록이 재계산되므로 실무상 거의 발생하지 않는다 — 재계산 사이의 좁은 창에서만 가능하고, 그 결과도 사용자가 Ctrl+Z로 되돌릴 수 있다. 별도 방어는 넣지 않는다.
- 저장 실패·재시도는 기존 `runBatch` 경로가 담당한다.

## 테스트

`tests/domain/weeklyLint.test.ts`에서 순수 함수를 검증한다.

- 중복: 같은 열 두 구분에 동일 줄 → 지적 1건, 뒤 구분에서 삭제하는 edit / 글머리·번호가 달라도 동일 판정 / 다른 열의 같은 줄은 미지적 / 3중 중복은 1건으로 묶임 / 빈 줄은 미지적
- 체번: 1,2,4 → 재번호 / 1,2,2 → 재번호 / 2,3,4 → 1,2,3 / 번호 줄 1개면 미지적 / `1)` 스타일 보존 / 번호 없는 줄 사이에 끼어도 순서 유지
- 정리: 줄 끝 공백 / 연속 빈 줄 / 전각 공백 / 글머리 기호 다수결 / 정리할 것 없으면 미지적
- 정렬 순서와 `id` 유일성

UI는 기존 `tests/ui` 관례를 따라 패널 렌더·적용 호출 여부 수준만 확인한다.

## 하지 않는 것

- LLM 기반 문장 품질 검토
- 일괄 자동 수정("모두 수정")
- 셀 안 인라인 밑줄 표시
- 같은 셀 안 중복 줄, 금주↔차주 중복, 내용↔이슈 중복 검사
- 빈 셀·미작성 구분 알림, 길이 경고, 날짜 표기 통일
- 점검 결과 저장·이력

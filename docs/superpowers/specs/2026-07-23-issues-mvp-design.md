# 이슈관리 MVP 스펙 — 프로젝트별 이슈 메뉴

> **상태:** 확정 스펙 (구현 전)
> **작성일:** 2026-07-23
> **결정 경로:** 브레인스토밍 문답 → 참조 설계([dflow-issue-management-design.md](../../design/dflow-issue-management-design.md)) 대조 → 4관점 병렬 감사(코드베이스 정합성·스코프·도메인/권한·UX) + 교차 검증 워크플로 → 사용자 확정
> **프레임:** 미니멀 안을 뼈대로 하고, 참조 설계에서 "소급 복원 불가 항목 + 검증된 관례 + 향후 접점의 전제"만 선별 수혈한다. 참조 설계의 나머지(링크·감사 이력·알림·보드 등)는 폐기가 아니라 Phase 2+ 백로그로 보존한다.

## 1. 확정 결정 요약

| 항목 | 결정 | 핵심 근거 |
|---|---|---|
| 이슈의 성격 | 프로젝트 리스크/장애물 (PM 관점) | 사용자 확정 |
| 메뉴 | 프로젝트별 `/p/[projectId]/issues` 하나만 | 사용자 확정 |
| 유형 필드 | 없음 (단일 개념) | type 컬럼은 후일 default와 함께 additive 추가 가능 — 소급 문제 없음 |
| 상태 | `open` / `in_progress` / `resolved` / `on_hold` 4개 | 프로덕션 40계정 중 27명(67.5%)이 pmo_admin — `해결 요청→PMO 검증 종료` 5단계는 직무분리 효과 없음 |
| 심각도 | `high` / `medium` / `low` | 사용자 확정 (표준형) |
| 조치 경과 | 단일 `resolution_note` 컬럼 | issue_updates 테이블은 순수 additive라 후일 도입 가능 |
| 화면 | 필터 테이블 + 모달 + KPI 3장 + `?focus=` 딥링크 | 회의 테이블 골격 + 공지 3모달·focus 구조 조립이 최소 코드 |
| 권한 | 회의 패턴 (멤버 등록 / 작성자·pmo_admin 수정·삭제 / 상태·담당자·조치메모는 멤버 전체) | 기존 컨벤션(meetings.ts:159) |
| 연동 | 1차는 독립 기능 | 사용자 확정 — 연동 자산은 §9 백로그 |
| 삭제 | hard delete (soft archive 불채택) | 회의·공지·회의록 관례와 정합 |
| 동시 편집 | 상태 변경 CAS (`where status=$from`, 0행 감지) | version 낙관잠금 불채택 — 기존 last-write-wins 관례 |

## 2. 데이터 모델 — `supabase/migrations/0040_issues.sql`

번호는 **0040** — 0039는 회의록 탐색기(`0039_minutes_explorer.sql`, 병렬 작업분)가 선점(2026-07-23 실측). 멱등 SQL + 롤백 파일(`0040_issues_rollback.sql`) 동봉. 적용은 Supabase Management API 경로(레포 관례 — db push 금지).

```sql
create table if not exists issues (
  id uuid primary key default gen_random_uuid(),
  issue_no bigint generated by default as identity unique,   -- 표시 "#12". 소급 발번 불가 → day-one 필수
  project_id uuid not null references projects(id) on delete cascade,
  title text not null,
  body text not null default '',
  status text not null default 'open'
    check (status in ('open','in_progress','resolved','on_hold')),
  severity text not null default 'medium'
    check (severity in ('high','medium','low')),
  assignee_member_id uuid,
  due_date date,
  resolution_note text not null default '',
  resolved_at timestamptz,                                    -- resolved 진입 시각. 소급 복원 불가 → day-one
  created_by uuid references auth.users(id) on delete set null,
  created_by_name text,                                       -- 탈퇴 후 표시용 스냅샷 (0013 회의 관례)
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),               -- AI 인덱스 신선도 가드의 필수 입력 (향후 접점 전제)
  -- 타 프로젝트 멤버 담당 혼입 DB 차단 (0032 근태 선례의 유니크 인덱스 재사용).
  -- 주의: 컬럼 지정 SET NULL 구문 필수 — 무지정 SET NULL은 NOT NULL project_id까지
  -- null 시도되어 멤버 삭제가 런타임 실패한다. 프로덕션 PG 17.6 실측으로 구문 사용 가능 확인.
  foreign key (assignee_member_id, project_id)
    references project_members(id, project_id)
    on delete set null (assignee_member_id)
);
create index if not exists issues_project_idx on issues(project_id, created_at desc);
```

**RLS** (헬퍼는 `app_role()` — 레포 정본):

| 정책 | 규칙 |
|---|---|
| select | authenticated 전체 (앱 관례) |
| insert | `created_by = auth.uid() and app_role() is not null` |
| update | `app_role() is not null` — 멤버 전체 백스톱. 필드 세분화(전체 편집 vs 진행 필드)는 액션 계층이 담당 |
| delete | `created_by = auth.uid() or app_role() = 'pmo_admin'` |

RLS는 행 단위라 열 단위 요구("상태·담당자는 멤버 누구나, 제목·내용은 작성자/pmo만")를 표현할 수 없다. DB는 "멤버면 수정 가능"까지 보장하고, 세분화는 서버 액션에서 fail-closed로 강제한다. 참조 설계의 컬럼 권한 매트릭스·전용 RPC·OLD/NEW 트리거는 불채택.

## 3. 도메인 — `src/lib/domain/issues.ts` (순수 함수, I/O 없음)

- `Issue` 타입 + `ISSUE_STATUSES` / `ISSUE_SEVERITIES` 상수 (DB에는 영문 코드만 저장 — i18n·봇 필터 전제)
- **전환 맵 상수** `STATUS_TRANSITIONS: Record<IssueStatus, IssueStatus[]>` — 단일 정본:
  - `open → in_progress | on_hold | resolved`
  - `in_progress → open | on_hold | resolved`
  - `on_hold → open | in_progress | resolved`
  - `resolved → open | in_progress` (재오픈)
  - 5번째 상태를 나중에 추가할 때 이 상수 + check 제약 변경으로 국한되도록 UI·액션 모두 이 맵만 참조한다.
- `resolved` 진입 시 `resolved_at = now()`, 이탈 시 `resolved_at = null` (액션에서 적용, 규칙은 도메인에 명시)
- `isOverdue(issue, today)` — 지연 = `due_date < today(Asia/Seoul) && status !== 'resolved'`
- `sortIssues` — 미해결 우선 → 지연 우선 → 심각도(high 먼저) → 목표일 오름차순(없으면 뒤) → 최신 등록순
- `summarizeIssues(issues, today)` — KPI 3장용 `{ open, inProgress, overdue }`
- `filterIssues(issues, { status, severity, mineOnly })` — 필터 칩 적용

## 4. 데이터 계층 — `src/lib/data/issues.ts`

- `getIssues(projectId)` — `cache()` 래핑, 담당자 이름은 `project_members` 조인, 실패 시 `[]` + `console.error` (읽기 계층 관례)
- "내 담당" 판정은 RLS가 아닌 표시 계층: `resolveMemberIds` 패턴(meetings 데이터 계층의 user_id + 정규화 email 이중 매칭 합집합)을 재사용해 세션 사용자의 member id 집합을 구하고 `assignee_member_id` 포함 여부로 판정. user_id 미연결 멤버의 fail-silent를 회피한다.

## 5. 서버 액션 — `src/app/actions/issues.ts`

회의 액션의 `{ ok, error }` 반환 관례. 전부 세션+멤버십 fail-closed. 각 쓰기 후 `revalidatePath('/p/[id]/issues')`.

| 액션 | 게이트 | 비고 |
|---|---|---|
| `createIssue(projectId, input)` | 멤버 전체 | `created_by_name` 스냅샷 저장 |
| `updateIssue(issueId, input)` | 작성자 또는 pmo_admin | 제목·내용·심각도·기한·담당자 전체 편집 |
| `updateIssueProgress(issueId, patch)` | 멤버 전체 | `status` / `assigneeMemberId` / `resolutionNote`만. 상태 변경은 전환 맵 검증 + CAS(`update … where id=$id and status=$from`) — 0행이면 "다른 사용자가 먼저 변경" 안내. resolved 진입/이탈 시 `resolved_at` 세팅/해제 |
| `deleteIssue(issueId)` | 작성자 또는 pmo_admin | hard delete |

입력 규칙: 제목 필수·trim·최대 200자, 내용·조치메모 각 최대 20,000자(참조 설계 §8과 동일), 담당자는 해당 프로젝트 멤버만(복합 FK가 DB에서 이중 방어), 과거 기한 허용(즉시 지연 표시로 보일 수 있음을 폼에서 안내).

## 9. 범위 제외 → Phase 2+ 백로그 (참조 설계에 보존)

참조 설계 [dflow-issue-management-design.md](../../design/dflow-issue-management-design.md)를 다음 항목의 설계 원천으로 유지한다(검증 결과 반영 개정본):

- `issue_links` (WBS·회의록 원문 앵커·`source_key` 중복 방지 — 위험 신호 키는 `risk-signal:{projectId}:{signal.id}`로 확정, 근거 목록 기반 키는 불안정하여 폐기)
- `issue_updates`(댓글형 조치 이력), `transitionIssue` RPC형 감사 이벤트(트리거 방식보다 actor 식별 우월)
- 영속 알림 테이블, 첨부파일 + Storage RLS, 보드 뷰, 유형 3종, `resolution_requested` 상태
- **주간시트 이슈 셀 → 이슈 등록 원클릭 승격** (회의→공지 승격 패턴 복제 — 이중 기입 문제의 해법, Phase 2 최우선 후보)
- 주간보고 삽입은 **시트 셀 텍스트로 저장** — 주간 PPT 파이프라인은 시트 셀만 읽으므로 PPT 개조 0 (이슈 테이블을 PPT가 직접 읽는 개조로 흐르지 않도록 명시)
- DK Bot: `BOT_DOMAINS`에 'issues' + 조회 도구 + `issueHref` 딥링크 빌더 / 대시보드 위험 신호 카드 "이슈로 등록" 프리필 / PM Assistant 브리핑에 미해결 이슈 신호 1줄
- 대회 데모 시드 스크립트 (운영 D-CUBE 쓰기 금지 — 전용 테스트 프로젝트에서만)

## 10. 수용 기준

- [ ] 사이드바·모바일 메뉴에서 이슈관리 진입 가능 (모바일 weekly 누락 drift도 해소됨)
- [ ] 멤버가 제목만으로 이슈를 등록할 수 있고 `#번호`가 동시 등록에서도 중복 없이 발급된다
- [ ] 상태·심각도·내 담당 필터, KPI 3장, 지연 강조가 정확하다
- [ ] 전환 맵에 있는 전환만 가능하고, 동시 상태 변경 시 한쪽은 안내를 받는다
- [ ] 진행 필드(상태·담당자·조치메모)는 멤버 전체, 전체 편집·삭제는 작성자/pmo_admin만 가능하다 (UI 노출 + 액션 게이트 이중)
- [ ] `?focus=` 딥링크로 상세 모달이 열리고 무효 id는 조용히 무시된다
- [ ] 타 프로젝트 멤버를 담당자로 지정할 수 없다 (DB 복합 FK)
- [ ] 담당 멤버 삭제 시 이슈는 유지되고 담당자만 비워진다
- [ ] 도메인 테스트·기존 스위트·빌드 그린
